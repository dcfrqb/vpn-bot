## Определение проблемы

Пользователи VPN-сервисов сталкиваются с необходимостью проходить сложные процедуры регистрации на веб-сайтах, вводить банковские карты и управлять подписками через веб-интерфейсы, что создает барьеры для быстрого доступа к услуге. Существующие решения требуют множества действий от пользователя и не обеспечивают удобного способа покупки и управления подпиской в привычной среде общения.

## Выработка требований

### Роли пользователей

1. **Пользователь** — покупатель VPN-подписки, который взаимодействует с системой через Telegram для приобретения и управления подпиской.
2. **Администратор** — управляет системой, просматривает статистику по пользователям и платежам, управляет настройками бота.

### User Stories

**US-1: Покупка VPN-подписки**
Как пользователь, я хочу выбрать тарифный план и оплатить VPN-подписку, чтобы получить доступ к VPN-сервису без необходимости регистрации на веб-сайте.

**US-2: Получение конфигурации после оплаты**
Как пользователь, я хочу автоматически получить ссылку на конфигурацию VPN после успешной оплаты, чтобы настроить VPN-клиент и начать использовать сервис.

**US-3: Просмотр статуса подписки**
Как пользователь, я хочу просматривать информацию о своей активной подписке, включая срок действия и тип тарифа, чтобы контролировать состояние подписки.

**US-4: Управление подпиской администратором**
Как администратор, я хочу просматривать статистику по пользователям, платежам и подпискам, чтобы анализировать работу системы и принимать управленческие решения.

### Оценка нагрузки

- **Предполагаемое количество пользователей:** 500-1 000 пользователей в сутки
- **Пиковая нагрузка:** 30–50 одновременных пользователей
- **Среднее количество транзакций:** 10–30 платежей в сутки
- **Соотношение операций:** 85% операций чтения (проверка статуса подписки, просмотр информации),
  15% операций записи (создание платежей, обновление подписок)
  
### Срок хранения данных

Все данные пользователей, платежей и подписок хранятся не менее **5 лет** для соответствия требованиям финансового учета, аудита и возможности восстановления истории транзакций.

## Архитектура и проектирование

### Архитектурные диаграммы (C4 Model)

#### Context Diagram (Уровень 1)

Система взаимодействует со следующими внешними акторами:

- **Пользователи Telegram** — взаимодействуют с системой через Telegram Bot API, отправляя команды и получая ответы
- **YooKassa** — внешний платежный провайдер, обрабатывающий платежи и отправляющий webhook-уведомления о статусе платежей
- **Remna API** — внешний сервис для управления VPN-подписками, создания пользователей и получения конфигураций VPN

Система обеспечивает связь между пользователями Telegram и внешними сервисами, обрабатывая платежи и управляя подписками.

#### Container Diagram (Уровень 2)

Система состоит из следующих контейнеров:

- **Telegram Bot** (Python, aiogram) — обрабатывает команды и сообщения от пользователей, отправляет ответы через Telegram Bot API, работает в режиме polling или webhook
- **FastAPI Webhook Server** (Python, FastAPI) — обрабатывает входящие webhook-уведомления от YooKassa о статусе платежей, обеспечивает REST API для внешних интеграций
- **PostgreSQL** — реляционная база данных для хранения данных пользователей, подписок, платежей и метаданных
- **Redis** (опционально) — используется для кэширования данных и временного хранения состояния (FSM storage)

Внешние сервисы:
- **YooKassa API** — обработка платежей
- **Remna API** — управление VPN-подписками и пользователями

### Характер нагрузки

**Соотношение операций чтения и записи:**
- **85% операций чтения** — проверка статуса подписки, получение информации о пользователе, просмотр статистики, чтение конфигураций
- **15% операций записи** — создание платежей, обновление статусов подписок, регистрация новых пользователей, обновление данных после webhook-уведомлений

**Объемы сетевого трафика:**
- **Входящий трафик:** ~3–8 МБ/день (webhook-уведомления от YooKassa, команды от пользователей через Telegram API)
- **Исходящий трафик:** ~15–30 МБ/день (ответы бота пользователям, отправка конфигураций VPN, запросы к внешним API)
  
*Примечание: Указанные значения являются оценочными и зависят от активности пользователей, частоты использования бота и объема передаваемых данных.*

**Объемы дискового хранилища:**
- **Данные пользователей:** ~5 МБ на ~50 000 уникальных пользователей за 5 лет (базовые данные профилей)
- **Платежи:** ~27 МБ на ~55 000 платежей за 5 лет (30 платежей/день × 365 дней × 5 лет, данные транзакций, метаданные)
- **Подписки:** ~22 МБ на ~55 000 подписок за 5 лет (конфигурации, статусы)
- **Логи:** ~100 МБ/месяц (логирование операций и ошибок)
- **Итого на 5 лет хранения:** ~54 МБ базовых данных + ~6 ГБ логов (при ротации)

### Схема базы данных

База данных PostgreSQL состоит из следующих основных таблиц:

**telegram_users** — пользователи Telegram
- `telegram_id` (BIGINT, PK) — уникальный идентификатор пользователя в Telegram
- `username`, `first_name`, `last_name` — данные профиля
- `remna_user_id` (FK) — связь с пользователем в Remna API
- `is_admin` (BOOLEAN) — флаг администратора
- `created_at`, `updated_at`, `last_activity_at` — метаданные

**remna_users** — пользователи из Remna API
- `remna_id` (VARCHAR, PK) — идентификатор пользователя в Remna API
- `username`, `email` — данные пользователя
- `raw_data` (JSON) — полные данные из API для гибкости
- `created_at`, `updated_at`, `last_synced_at` — метаданные синхронизации

**subscriptions** — подписки на VPN
- `id` (INTEGER, PK) — внутренний идентификатор
- `telegram_user_id` (BIGINT, FK) — связь с пользователем Telegram
- `remna_user_id` (VARCHAR, FK) — связь с пользователем Remna
- `plan_code` (VARCHAR) — код тарифа (basic, premium, trial)
- `active` (BOOLEAN) — статус активности
- `valid_until` (TIMESTAMP) — срок действия подписки
- `remna_subscription_id` (VARCHAR) — идентификатор подписки в Remna API
- `config_data` (JSON) — данные конфигурации VPN

**payments** — платежи
- `id` (INTEGER, PK) — внутренний идентификатор
- `telegram_user_id` (BIGINT, FK) — связь с пользователем
- `external_id` (VARCHAR, UNIQUE) — идентификатор платежа во внешней системе (YooKassa)
- `provider` (VARCHAR) — провайдер платежей (yookassa, crypto)
- `amount` (NUMERIC) — сумма платежа
- `currency` (VARCHAR) — валюта
- `status` (VARCHAR) — статус (pending, succeeded, canceled, failed)
- `subscription_id` (INTEGER, FK) — связь с созданной подпиской
- `description`, `payment_metadata` (JSON) — дополнительные данные
- `created_at`, `updated_at`, `paid_at` — временные метки

**Дополнительные таблицы:**
- `squads` — сквады из Remna API (опционально)
- `nodes` — ноды из Remna API (опционально)

**Обоснование структуры:**
- Нормализованная схема для избежания дублирования данных
- Индексы на часто используемых полях (`telegram_id`, `external_id`, `status`, `active`) для оптимизации запросов
- Foreign Key ограничения для обеспечения целостности данных
- JSON поля для гибкого хранения дополнительных данных без изменения схемы
- Каскадное удаление связанных записей при удалении пользователя
- Временные метки для аудита и аналитики

## Нефункциональные требования

### Производительность

- **Время отклика бота:** менее 2 секунд для обработки команды пользователя и отправки ответа
- **Время обработки webhook:** менее 500 мс для обработки webhook-уведомления от YooKassa и обновления статуса платежа
- **Производительность БД:** поддержка до 1000 запросов в секунду при текущей нагрузке

### Доступность

- **Uptime:** 99.5% доступности системы (допускается до 43.8 часов простоя в год)
- **Обработка ошибок:** система должна корректно обрабатывать временные сбои внешних API (YooKassa, Remna API) с повторными попытками

### Масштабируемость

- **Горизонтальное масштабирование:** архитектура позволяет запускать несколько инстансов бота и webhook-сервера
- **Асинхронная обработка:** все взаимодействия с внешними API выполняются асинхронно для предотвращения блокировок

### Безопасность

- Все секретные данные (токены, API ключи) хранятся в переменных окружения, не коммитятся в репозиторий
- Используется HTTPS для всех webhook-соединений
- Валидация всех входящих данных от пользователей и внешних сервисов
- Логирование всех критических операций для аудита

## Быстрое развертывание

### 1. Клонирование репозитория

```bash
git clone <repository-url>
cd VPN/TGBot
```

### 2. Настройка конфигурации

Скопируйте файл конфигурации:

```bash
cp config.example.env .env
```

Отредактируйте `.env` файл и заполните обязательные параметры:

```env
# Обязательные параметры
BOT_TOKEN=your_telegram_bot_token
ADMINS=123456789,987654321

# YooKassa (для платежей)
YOOKASSA_SHOP_ID=your_shop_id
YOOKASSA_API_KEY=your_api_key
YOOKASSA_RETURN_URL=https://t.me/your_bot_username?start=payment_success
YOOKASSA_WEBHOOK_SECRET=your_webhook_secret

# Remna API (для VPN)
REMNA_API_BASE=https://your-remna-api.com
REMNA_API_KEY=your_remna_api_key
```

**Примечание:** База данных и Redis настраиваются автоматически через Docker Compose. Если нужно использовать внешние сервисы, укажите `DATABASE_URL` и `REDIS_URL` в `.env`.

### 3. Запуск одной командой

Рекомендуемый способ — использовать скрипт `run_all.sh`, который выполняет все необходимые операции в правильном порядке:

```bash
# Выполнить все: сборка -> unit-тесты -> интеграционные тесты -> запуск
./run_all.sh all
```

Команда `all` последовательно выполняет:
1. **Сборку проекта** — создание Docker образов для всех сервисов
2. **Запуск unit-тестов** — проверка изолированной бизнес-логики
3. **Запуск интеграционных тестов** — проверка полных пользовательских сценариев
4. **Запуск приложения** — после успешного прохождения всех тестов запрашивается подтверждение для запуска приложения

**Примечание:** Команда `all` запрашивает подтверждение перед запуском приложения. Для автоматического запуска без подтверждения используйте отдельные команды:

```bash
# 1. Сборка
./run_all.sh build

# 2. Все тесты (unit + integration)
./run_all.sh test-all

# 3. Запуск приложения
./run_all.sh run
```

Или выполните тесты и запуск отдельно:

```bash
# Unit-тесты
./run_all.sh test-unit

# Интеграционные тесты
./run_all.sh test-integration

# Запуск приложения
./run_all.sh run
```

### 4. Запуск через Docker Compose напрямую

```bash
docker compose up -d
```

Бот автоматически:
- Создаст и запустит PostgreSQL и Redis
- Применит миграции базы данных (при запуске бота)
- Запустит Telegram бота
- Запустит FastAPI webhook сервер для обработки платежей

### 5. Проверка работы

Проверьте логи:

```bash
docker compose logs -f bot
```

Или проверьте статус всех сервисов:

```bash
docker compose ps
```

## Масштабирование

При росте нагрузки в 10 раз (до 5 000–10 000 пользователей в сутки, до 300–500 одновременных пользователей, до 100–300 платежей в сутки) система масштабируется следующим образом:

### Горизонтальное масштабирование

**1. Telegram Bot:**
- Запуск нескольких инстансов бота с использованием Redis для синхронизации состояния между инстансами
- Переход с polling на webhook режим для снижения нагрузки на Telegram API и более эффективной обработки обновлений
- Использование очередей сообщений для распределения нагрузки между инстансами

**2. FastAPI Webhook Server:**
- Запуск нескольких инстансов за load balancer (nginx или аналогичный)
- Stateless архитектура позволяет легко масштабировать без необходимости синхронизации состояния между инстансами
- Асинхронная обработка webhook-уведомлений для предотвращения блокировок

**3. PostgreSQL:**
- Настройка репликации: 1 master для записи + 2–3 read replicas для чтения
- Read-запросы (85% нагрузки) направляются на read replicas
- Write-запросы (15% нагрузки) идут на master
- Партиционирование таблицы `payments` по дате для оптимизации работы с большими объемами исторических данных
- Настройка connection pooling для эффективного использования соединений

**4. Redis:**
- Использование Redis Cluster для распределения нагрузки и данных между несколькими узлами
- Настройка Redis Sentinel для обеспечения высокой доступности и автоматического failover

### Вертикальное масштабирование

- Увеличение ресурсов контейнеров (CPU, RAM) для обработки большего количества одновременных запросов
- Увеличение ресурсов БД (больше RAM для кэша, больше CPU для обработки запросов)

### Оптимизация

- **Кэширование:** кэширование часто запрашиваемых данных (информация о подписках пользователей, статистика) в Redis для снижения нагрузки на БД
- **Асинхронная обработка:** тяжелые операции (синхронизация с Remna API, отправка уведомлений) выполняются асинхронно через очереди задач
- **Батчинг:** группировка запросов к внешним API (Remna API) для снижения количества HTTP-запросов
- **Мониторинг:** настройка мониторинга производительности и алертинга для раннего обнаружения проблем и узких мест

## Безопасность

- Все секретные данные хранятся в `.env` файле (не коммитится в Git)
- Используется HTTPS для webhook'ов
- Валидация всех входящих данных
- Логирование всех операций

## Тестирование

Проект включает unit-тесты и интеграционные тесты, покрывающие пользовательские истории и бизнес-логику системы.

### Unit-тестирование

В проекте реализовано **32 unit-теста**, которые проверяют изолированную бизнес-логику без зависимости от внешних сервисов:

- **Тесты платежей** (`test_payments.py`) — 13 тестов:
  - Создание платежей через YooKassa API
  - Обработка webhook-уведомлений о статусе платежей
  - Проверка статуса платежей
  - Обработка успешных платежей и создание подписок
  - Обработка ошибок и граничных случаев

- **Тесты статистики** (`test_stats.py`) — 4 теста:
  - Получение общей статистики
  - Списки пользователей с пагинацией
  - Списки платежей с фильтрацией

- **Тесты форматирования** (`test_subscription_formatter.py`) — 7 тестов:
  - Форматирование информации о подписках
  - Обработка различных состояний подписок (активная, истекшая, без срока)

- **Тесты подписок** (`test_subscriptions.py`) — 4 теста:
  - Проверка истечения подписок
  - Получение подписок, истекающих скоро
  - Отправка уведомлений об истечении

- **Тесты пользователей** (`test_users.py`) — 4 теста:
  - Создание и обновление пользователей
  - Получение активных подписок пользователей

**Особенности unit-тестов:**
- Используют моки (`unittest.mock`, `AsyncMock`, `MagicMock`) для изоляции от внешних зависимостей
- Не требуют подключения к реальной базе данных (используют in-memory SQLite или моки сессий)
- Не зависят от внешних API (Telegram API, YooKassa, Remna API)
- Проверяют реальную бизнес-логику, не являются тривиальными

### Интеграционное тестирование

В проекте реализовано **4 интеграционных теста** (`tests/integration/test_user_flow.py`), которые проверяют полные пользовательские сценарии end-to-end:

1. **Регистрация пользователя и создание пробной подписки** (`test_user_registration_flow`):
   - Создание нового пользователя в базе данных
   - Автоматическое создание пробной подписки
   - Проверка корректности сохранения данных в БД

2. **Создание платежа, оплата и создание подписки** (`test_payment_and_subscription_flow`):
   - Создание платежа через YooKassa API
   - Сохранение платежа в базе данных
   - Обработка успешного платежа
   - Создание подписки после оплаты
   - Отправка уведомления пользователю

3. **Истечение подписки** (`test_subscription_expiration_flow`):
   - Создание подписки с истекающим сроком
   - Автоматическое обновление статуса при истечении
   - Проверка корректности обработки истекших подписок

4. **Работа с несколькими подписками пользователя** (`test_user_with_multiple_subscriptions`):
   - Создание нескольких подписок для одного пользователя
   - Проверка получения активной подписки
   - Проверка корректности работы с историей подписок

**Особенности интеграционных тестов:**
- Используют реальную PostgreSQL базу данных (тестовую)
- Проверяют взаимодействие нескольких компонентов системы (БД, сервисы, бизнес-логика)
- Покрывают полноценные пользовательские истории от начала до конца
- Изолированы друг от друга (каждый тест использует чистую БД)

### Запуск тестов одной командой

## Сборка и запуск

Проект полностью автоматизирован для сборки, тестирования и запуска через Docker. Все операции выполняются одной командой через скрипт `run_all.sh`.

### Выполнение всех операций одной командой

Для выполнения всех этапов разработки (сборка → unit-тесты → интеграционные тесты → запуск) используйте:

```bash
./run_all.sh all
```

Эта команда последовательно выполняет:
1. **Сборку проекта** — создание Docker образов для всех сервисов (bot, webhook-api)
2. **Запуск unit-тестов** — проверка изолированной бизнес-логики (32 теста)
3. **Запуск интеграционных тестов** — проверка полных пользовательских сценариев (4 теста)
4. **Запуск приложения** — после успешного прохождения всех тестов запрашивается подтверждение для запуска приложения

**Примечание:** Команда `all` запрашивает подтверждение перед запуском приложения для возможности проверки результатов тестов. Это соответствует требованию курсовика о выполнении всех операций одной командой.

### Выполнение операций по отдельности

Если необходимо выполнить операции по отдельности:

```bash
# 1. Сборка Docker образов
./run_all.sh build

# 2. Unit-тесты
./run_all.sh test-unit

# 3. Интеграционные тесты
./run_all.sh test-integration

# 4. Все тесты
./run_all.sh test-all

# 5. Запуск приложения
./run_all.sh run
```

### Запуск через Docker Compose напрямую

Для запуска приложения без тестов можно использовать Docker Compose напрямую:

```bash
docker compose up -d
```

При запуске автоматически:
- Создаются и запускаются контейнеры PostgreSQL и Redis
- Применяются миграции базы данных
- Запускается Telegram бот
- Запускается FastAPI webhook сервер для обработки платежей

### Требования

Для работы проекта требуется только:
- **Docker** — для контейнеризации и изоляции сервисов
- **Bash** — для выполнения скрипта `run_all.sh`

Все зависимости (Python, PostgreSQL, Redis) устанавливаются и настраиваются автоматически через Docker Compose.
