# Документация по тестам

## Обзор

Добавлен полный набор тестов, покрывающий:
- Сборку проекта и импорт модулей
- Базовую логику бота
- Бизнес-логику подписок
- Строгие сценарии (force_remna)
- Регрессию по производительности

## Структура тестов

### 1. `test_build.py` - Тесты сборки и импорта

**Назначение:** Проверяют, что проект собирается без ошибок и все модули импортируются корректно.

**Покрытие:**
- Импорт ключевых модулей (bot, sync_service, remna client, cache layer)
- Импорт handlers, repositories, models
- Валидация конфигурации
- Проверка на циклические зависимости
- Проверка структуры проекта

**Запуск:**
```bash
PYTHONPATH=src pytest tests/test_build.py -v
```

### 2. `test_sync_service.py` - Юнит-тесты бизнес-логики

**Назначение:** Защищают логику источника истины (SyncService).

**Покрытие:**
- Пользователь найден в Remna → active
- Пользователь найден в Remna → expired
- Пользователь не найден → none
- `force_remna=True`: кэш НЕ используется, fallback НЕ используется
- Remna недоступна: обычный режим → fallback, force_remna → исключение

**Запуск:**
```bash
PYTHONPATH=src pytest tests/test_sync_service.py -v
```

### 3. `test_handlers_start.py` - Тесты логики бота (/start)

**Назначение:** Гарантируют корректное поведение UX для команды /start.

**Покрытие:**
- Быстрый ответ
- Не падает при отсутствии подписки
- Кнопка "Обновить" вызывает force_remna
- Кнопка "Обновить" возвращает актуальный статус

**Запуск:**
```bash
PYTHONPATH=src pytest tests/test_handlers_start.py -v
```

### 4. `test_handlers_friend.py` - Тесты логики бота (/friend)

**Назначение:** Гарантируют корректное поведение строгих сценариев для команды /friend.

**Покрытие:**
- /friend всегда выполняет force_remna
- /friend не использует кэш или БД
- Подписка есть → запрос запрещён
- Подписки нет → запрос разрешён

**Запуск:**
```bash
PYTHONPATH=src pytest tests/test_handlers_friend.py -v
```

### 5. `test_force_remna.py` - Строгие сценарии

**Назначение:** Защита от логических багов и защита доверия к данным.

**Покрытие:**
- force_remna при удалённой подписке → Remna → none
- Бот не считает подписку активной, если её нет в Remna
- force_remna НИКОГДА не использует кэш
- force_remna НИКОГДА не использует fallback
- Защита от возврата устаревших active подписок

**Запуск:**
```bash
PYTHONPATH=src pytest tests/test_force_remna.py -v
```

### 6. `test_performance.py` - Регрессионные тесты производительности

**Назначение:** Ловят деградацию скорости при будущих изменениях.

**Покрытие:**
- SyncService без Remna: выполнение < 5 мс
- /start handler при cache hit: выполнение < 100 мс
- Создание клавиатур: не превышает заданный порог
- SyncService fallback: выполнение < 10 мс

**Запуск:**
```bash
PYTHONPATH=src pytest tests/test_performance.py -v
```

## Запуск всех тестов

```bash
# Все тесты
PYTHONPATH=src pytest tests/ -v

# Только unit тесты (без интеграционных)
PYTHONPATH=src pytest tests/ -v -m "not integration"

# С покрытием кода
PYTHONPATH=src pytest tests/ --cov=src/app --cov-report=html --cov-report=term -m "not integration"
```

## Важные принципы

1. **Не используются реальные сервисы:** Все тесты используют моки для Remna API, Redis, PostgreSQL
2. **Не используются sleep:** Тесты не зависят от времени ожидания
3. **Не проверяется Telegram API напрямую:** Используются моки для aiogram
4. **Производительность измеряется через time.monotonic:** Для стабильности тестов

## Защита от регрессий

Тесты защищают от:
- Ошибок импорта и циклических зависимостей
- Логических багов в бизнес-логике
- Возврата устаревших данных о подписках
- Деградации производительности
- Нарушения строгих сценариев (force_remna)

## CI/CD

Тесты автоматически запускаются в CI при каждом push/PR в ветку main (см. `.github/workflows/python-ci.yml`).
